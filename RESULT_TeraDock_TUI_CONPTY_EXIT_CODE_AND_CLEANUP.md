# TeraDock TUI ConPTY Exit Code and Cleanup Result

Date: 2026-06-17

## Summary

- `exit_code=255` is not generated by session metadata classification. It comes
  from the ConPTY child wait path, via `portable-pty` `ExitStatus::exit_code()`.
- The saved session `sl_u5bnwu44` is not normal `exit` evidence. Its log tail
  ended with an SSH client disconnect/reset message, and metadata recorded
  `status=completed_nonzero`, `exit_code=255`.
- Source inspection found no normal-path child kill before TUI restore. The TUI
  suspends, runs the ConPTY runner, waits for completion, and only then restores.
- Failure/abort cleanup evidence was too weak before this change because the
  wait thread only reported "done" to cleanup code. This was fixed by returning
  a structured child wait outcome.
- `auto -> conpty` was not promoted.

## Phase 1 Findings

`exit_code=255` source:

- `crates/core/src/conpty.rs` waits on the ConPTY child and converts
  `portable-pty` status through `conpty_exit_code`.
- `crates/core/src/session_log.rs` stores that integer and maps `Some(255)` to
  `completed_nonzero`.
- No metadata code maps a normal completion to `255`.

Normal `exit` and `255` risk:

- A normal remote `exit` should be recorded as the actual SSH client exit code,
  normally `0`.
- If normal `exit` still records `255`, the SSH client itself is exiting `255`
  or the PTY/session shutdown is causing SSH to disconnect before normal
  completion. The existing `sl_u5bnwu44` evidence does not prove that case.

PTY close, child kill, and abort separation:

- Child kill is only called on startup timeout, user abort, output bridge error,
  input bridge error, or child wait error.
- The `ChildExited` path does not call child kill.
- TUI restore happens after the suspended SSH session runner returns.

Metadata completion:

- `exit_code=0` maps to `status=completed`.
- Nonzero exit codes map to `status=completed_nonzero`.
- Runner failures use explicit `failed` or `aborted` metadata with failure
  phase/reason.

Child cleanup evidence:

- Before this change, success cleanup was inferred from receiving
  `ChildExited`, but failure/abort cleanup did not preserve the wait result.
- After this change, debug output can show `child wait joined: code ...` and
  `no child process remains` when the child wait thread confirms exit.

## Code Changes

- Added a structured `ConptyWaitOutcome` for child wait completion.
- Made normal completion require a confirmed child wait outcome.
- Preserved the child exit code on runner failure/abort when the wait thread
  can confirm it.
- Passed failure-path exit codes into ConPTY metadata from both CLI and TUI
  callers.
- Updated debug cleanup messages:
  - `input bridge joined`
  - `pty handles dropped`
  - `child wait joined: code <code>`
  - `output reader joined`
  - `no child process remains`

No SSH command line, auth args, private key path, secret, token, or environment
value is added to metadata or debug output.

## Expected Case Handling

- Remote `exit`: `status=completed`, `exit_code=0`.
- Remote `exit 7`: `status=completed_nonzero`, `exit_code=7`.
- Ctrl-C abort: `status=aborted`; an exit code is recorded only if the child
  wait thread confirms one.
- Bad host/auth failure: `completed_nonzero` when `ssh.exe` exits with a
  nonzero code after running; `failed` when the ConPTY runner fails in setup,
  startup timeout, bridge, or wait handling.
- Timeout: `status=failed`, failure phase/reason recorded, child cleanup must
  be confirmed by manual process snapshot and debug output.
