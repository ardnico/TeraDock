# Release Artifact Validation

This document defines the v0.1.0 artifact validation pass. Do not push the
production `v0.1.0` tag until the checks below pass.

## 1. Validation strategy

Use the release workflow in this order:

1. Run `workflow_dispatch` with a validation version such as
   `0.1.0-rc-artifact`. This builds Linux and Windows artifacts and uploads
   them as GitHub Actions artifacts only. It does not create a GitHub Release.
2. If a release-like GitHub Release page must be tested, push an RC tag such as
   `v0.1.0-rc1`. The workflow marks `*-rc*` tags as prereleases.
3. Avoid test tags unless needed for GitHub tag behavior. Tags containing
   `-test` build artifacts but skip GitHub Release creation.
4. Push `v0.1.0` only after Actions artifacts, install smoke tests, release
   notes, and README instructions are verified.

Before the production release, verify:

- The release workflow completes on Linux and Windows.
- The expected artifacts and checksum files are present.
- Install and smoke tests pass on representative Linux and Windows machines.
- Release notes, README installation steps, and known limitations match the
  actual artifacts.
- No logs or recent-operation output expose secrets, auth arguments, private
  key paths, or full SSH command strings.

## 2. Expected artifacts

Actions artifact groups:

- `linux-artifacts`
- `windows-artifacts`

Expected files:

- Linux portable archive: `td-<version>-linux-x86_64.tar.gz`
- Debian package: generated by `cargo deb`; name includes package name, Cargo
  version, and Debian architecture.
- RPM package: generated by `cargo generate-rpm`; name includes package name,
  Cargo version, release, and RPM architecture.
- Windows installer: `td-<version>-windows-x86_64-setup.exe`
- Linux checksums: `SHA256SUMS-linux-x86_64`
- Windows checksums: `SHA256SUMS-windows-x86_64`

For a production tag release, these same files are uploaded as GitHub Release
assets. The tar.gz and Windows installer use the tag-derived version label. The
deb and rpm package metadata comes from the Cargo package version.

Checksum verification:

```bash
sha256sum -c SHA256SUMS-linux-x86_64
```

PowerShell:

```powershell
Get-FileHash .\td-0.1.0-windows-x86_64-setup.exe -Algorithm SHA256
Get-Content .\SHA256SUMS-windows-x86_64
```

## 3. Linux tar.gz smoke test

Use a clean temporary directory when possible.

```bash
tar -xzf <artifact>.tar.gz
cd td-<version>-linux-x86_64
./td --help
./td doctor
./td init --with-samples
./td profile list
./td config keys
./td recent --json
```

## 4. deb smoke test

```bash
sudo apt install ./<artifact>.deb
td --help
td doctor
td init --with-samples
td profile list
td recent --json
sudo apt remove td
```

## 5. rpm smoke test

```bash
sudo dnf install ./<artifact>.rpm
td --help
td doctor
td init --with-samples
td profile list
td recent --json
sudo dnf remove td
```

If no `dnf` environment is available, record that constraint in the release
validation notes and treat rpm install verification as incomplete.

## 6. Windows installer smoke test

Install the generated `.exe` on a clean Windows test machine or VM.

PowerShell:

```powershell
td.exe --help
td.exe doctor
td.exe init --with-samples
td.exe profile list
td.exe recent --json
```

Confirm:

- The installer places `td.exe` where expected.
- `td.exe` is available on `PATH`, or the release notes clearly state that it
  must be run from the install directory.
- Uninstall succeeds and removes installed program files.
- `td.exe doctor` output is clear when OpenSSH client is missing.
- `td.exe doctor` output is clear when OpenSSH client is present.
- Non-TTY `td.exe ui` exits with a clear error.

## 7. TUI manual smoke test

Run `td ui` in an interactive terminal and verify:

- The TUI starts and redraws correctly.
- `/` search works.
- Type, group, danger, tag, and query filters work.
- Details view opens.
- `s` opens an interactive SSH session for an SSH profile.
- Critical SSH-session confirmation requires typing the profile id.
- Non-SSH profiles do not open an SSH session.
- Exiting SSH returns to the TUI.

Do not run the SSH-session test against a production host. Use a disposable
test profile.

## 8. Security checks

Verify:

- Logs do not contain secrets, passwords, auth args, private key paths, or full
  SSH command strings.
- `td recent --json` does not expose excessive connection or credential data.
- FTP cannot be used without both `allow_insecure_transfers=true` and
  `--i-know-its-insecure`.
- Critical profile confirmation is enforced for connect, exec, run, transfer,
  config apply, and TUI SSH sessions.
- Default export excludes decrypted secret values.

## 9. Release go/no-go criteria

Go:

- `cargo fmt --check` passes.
- `cargo test` passes.
- `cargo clippy --all-targets --all-features -- -D warnings` passes.
- `cargo build -p td --release --locked` passes.
- Linux and Windows artifact generation succeeds.
- Checksums are generated and match downloaded artifacts.
- Install smoke tests pass.
- README instructions match the actual artifacts.
- Known limitations are documented.

No-Go:

- Any required artifact fails to build.
- An artifact cannot be installed.
- `td --help` cannot run after install or extraction.
- Database initialization fails.
- The TUI cannot return after an interactive SSH session.
- Secrets, passwords, auth args, private key paths, or full command strings
  appear in logs or recent-operation output.
- Release notes describe behavior or artifacts that do not exist.
